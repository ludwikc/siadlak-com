# Image Optimization Pipeline — Design Spec

**Date:** 2026-03-28
**Status:** Approved

---

## Problem

- 35 images in `public/lovable-uploads/` (PNG/JPEG, UUID filenames)
- 44 images in `src/assets/` (PNG, WebP, GIF)
- ~270 `<img>` tags across 30 files
- No AVIF/WebP `<picture>` elements
- No `loading="lazy"`
- No `fetchpriority`
- Total unoptimized image payload: ~75MB

## Goals

- Fastest possible load times for visitors (Vercel static hosting)
- Clean, maintainable code
- Compatible with ongoing Lovable.dev co-management

## Non-Goals

- CSS `background-image` references (18 found) — no `<picture>` equivalent
- GIF files (e.g. `I-know-kung-fu.gif`) — leave as-is to preserve animation
- Archive pages under `src/pages/archive/` — dead code, skip
- CDN/infrastructure changes

---

## Architecture

### Two-Track System

**Track A — `src/assets/` (Claude Code territory)**

Images imported as ES modules. `vite-imagetools` processes them at build time via Sharp, generating AVIF/WebP/PNG variants at 400/800/1200px widths with hashed filenames.

**Track B — `public/lovable-uploads/` (Lovable.dev territory)**

This directory is left exactly as Lovable expects it. A standalone script `scripts/optimize-images.ts` runs Sharp against it to generate `.webp` and `.avif` siblings alongside each original. The script is idempotent — safe to re-run after any Lovable session that adds new images.

Source path derivation: given `/lovable-uploads/foo.png`, the `<OptimizedImage>` component constructs `/lovable-uploads/foo.webp` and `/lovable-uploads/foo.avif` automatically.

---

## Component: `<OptimizedImage>`

**Location:** `src/components/ui/OptimizedImage.tsx`

```tsx
type OptimizedImageProps = {
  src: string | { avif: string; webp: string; fallback: string }
  alt: string
  width: number
  height: number
  priority?: boolean
  className?: string
  sizes?: string  // e.g. "(max-width: 768px) 100vw, 50vw"
}
```

**Behaviour:**
- `src` as string (Track B): derives `.webp` and `.avif` source paths from the original
- `src` as object (Track A): uses pre-built sources from `vite-imagetools`
- `priority={true}`: sets `fetchpriority="high"` + `loading="eager"` — for above-the-fold/LCP images
- Default: `loading="lazy"`
- `sizes` defaults to `"100vw"` if omitted; callers should provide a real value for non-full-width images
- `width` and `height` are required to prevent layout shift (CLS)

Renders a `<picture>` element with source order: AVIF → WebP → original format fallback.

---

## Script: `scripts/optimize-images.ts`

- Scans `public/lovable-uploads/` and `public/` for PNG/JPG/JPEG files
- Skips files that already have a `.webp`/`.avif` sibling (idempotent)
- Generates:
  - WebP at quality 85
  - AVIF at quality 70
- Outputs alongside originals — no path changes, no reference updates needed
- Registered as `npm run optimize-images` in `package.json`

---

## Vite Configuration

Add `vite-imagetools` to `vite.config.ts`:
- Output formats: AVIF, WebP, original
- Responsive widths: 400, 800, 1200px
- Applied to all `src/assets/` image imports

## `fetchpriority` Strategy

Images marked `priority`:
- Hero image on `Index.tsx`
- Hero image on `About.tsx`
- Hero images on `WebinarMeskiKompas.tsx`, `WebinarKodKapitana.tsx`, `WebinarDepresja.tsx`
- Hero images on `MeskiKompas.tsx`, `SilnaGlowa.tsx`, `UwazneZycie.tsx`

All other images: `loading="lazy"` (default, no explicit `fetchpriority`).

---

## Migration Steps

1. **Install & configure `vite-imagetools`** — update `vite.config.ts`
2. **Run `optimize-images` script** — generate WebP/AVIF for all `public/` images
3. **Build `<OptimizedImage>` component** — `src/components/ui/OptimizedImage.tsx`
4. **Audit all `<img>` usages** — classify as Track A or B, identify priority/LCP candidates
5. **Replace `<img>` tags** — file by file across all 30 files
6. **Verify build** — `npm run build`, confirm AVIF/WebP in `dist/`

---

## Ongoing Maintenance

After any Lovable session that adds new images:
```
npm run optimize-images
```

That is the only recurring cost. New `<img>` tags generated by Lovable will render without AVIF/WebP until a Claude Code pass replaces them with `<OptimizedImage>` — acceptable under the mixed workflow.
